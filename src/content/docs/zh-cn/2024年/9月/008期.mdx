## 随便聊聊

![](https://github.com/binarycoder777/personal-pic/blob/main/image.png?raw=true)

许久没玩游戏了，最近一周下了班回家就是玩当前很火的《黑神话·悟空》，玩了30来个小时也算是玩通关了。当然，我想在此说的是，劳逸结合，适度而可，玩这个游戏有些时候确实是着相了。期间为了打BOSS熬夜到一点多钟，也在网上找攻略，为了搜集一些物品到处搜刮，很喜欢里面的一句话：只怕你聋了耳又乱了心，越是贪图完满，越是搜不干净。

是啊，很多时候我们在处理事情时，往往想要把所有的细节都做到极致，追求一种绝对的完满。然而，这种心态可能会导致我们过于分散精力，就像在一个杂乱的空间中，想要一次性把所有东西都整理得完美无缺，结果却可能顾此失彼。比如在项目开发中，如果一味追求各个环节的完美，可能会忽略项目的整体进度和关键节点，反而影响最终的交付。

当我们不被 “贪图完满” 的心态所驱使时，我们的内心会更加平静。不会因为一点瑕疵或者不如意就焦虑不安，而是能够以一种平和的心态去面对生活中的种种。就像欣赏一幅画作，我们能够欣赏它整体的美，而不是只盯着那几处可能并不完美的笔触。你认为呢？

### [可扩展性是一个被低估的代码度量](https://morizbuesing.com/blog/greppability-code-metric/)

![](https://morizbuesing.com/_astro/scott-blake-bGmx6NxEVAU-unsplash.U-9YLhUu_1aFPNg.webp)

作者指出在维护不熟悉的代码库时，代码的可搜索性（greppability）是一个被低估的指标。为保持代码库可搜索，作者提出了一些规则：不要拆分标识符，动态构建标识符不利于维护；在整个代码栈中对事物使用相同的名称，不要在应用程序边界处重命名字段；遵循“扁平优于嵌套”原则，例如在设置翻译文件和 React 组件结构时，扁平的结构更便于搜索。文章强调了这些规则对提高代码可维护性的重要性。

### [黑客攻击虚拟电厂](https://rya.nc/vpp-hack.html)

作者在家里安装了 GivEnergy 的太阳能电池板和电池存储系统，试用其 API 生成的 JWT 令牌，发现是 512 位 RSA 签名，这种密钥易被破解。作者通过数学计算和工具恢复模量、破解钥匙，成功获取并修改 API 令牌，能访问任意账户信息。向厂商报告后，对方迅速修复，切换为 4096 位 RSA 密钥。作者指出不应期望开发者了解 512 位 RSA 的不安全，而应依靠专业的加密库，且自己正努力推动主要加密库放弃对 512 位 RSA 的支持。

### [在没有数据库的情况下构建高可用性 Web 服务](https://blog.screenshotbot.io/2024/08/10/building-a-highly-available-web-service-without-a-database/)

文章主要介绍了一种新的 Web 服务架构：

- 探讨阶段：初创公司时，可将 Web 服务和数据库实例合一，内存作为数据库，定期拍摄内存快照并记录事务日志以恢复。
- 扩展阶段：吸引大客户需要高可用性时，利用 Raft 共识协议在多台机器上复制事务日志，实现高度可用。
- 提取阶段：公司发展壮大后，可通过分片解决瓶颈。
  
作者还介绍了 Screenshotbot 所使用的技术栈及相关优势，认为这种架构适合新创公司，并希望更多公司采用。
